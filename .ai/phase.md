# Development Phases (phase.md)

이 문서는 AI 코딩 에이전트가 순차적으로 프로젝트를 빌드하기 위한 단계별 지침서입니다. 각 Phase가 완전히 작동하는지 테스트한 후 다음 Phase로 넘어가야 합니다.

## Phase 1: 프로젝트 초기화 및 인증 UI 구축

- **작업 내용:**
  1. Next.js (TypeScript, Tailwind CSS) 프로젝트 세팅.
  2. 환경 변수(`.env`)에 공통 접속용 비밀번호(`APP_PASSWORD`) 설정.
  3. **로그인 페이지(`app/page.tsx`)** 구현: 비밀번호 입력 폼 구성.
  4. 인증 로직: 입력값이 맞으면 `localStorage`에 인증 상태를 저장하고 `/menu`로 라우팅.
- **검증:** 브라우저에서 비밀번호를 입력하고 메인 메뉴로 정상 이동하는지, 새로고침 시 로그인이 유지되는지 확인.

## Phase 2: 메인 메뉴 및 바둑판 기본 렌더링 (✅ 완료)

- **작업 내용:**
  1. **메인 메뉴(`app/menu/page.tsx`)** 구현: '컴퓨터와 대결', '사활 퀴즈' 버튼 및 바둑판 크기(9x9, 13x13, 19x19) 선택 UI.
  2. 선택한 설정값을 쿼리 파라미터나 전역 상태로 저장하여 게임 화면(`/game`)으로 전달.
  3. `Tenuki.js` (최종적으로 `Sabaki` 생태계로 교체)를 연동하여 `app/game/page.tsx`에 빈 바둑판 렌더링.
- **검증:** 선택한 사이즈의 바둑판이 화면 중앙에 반응형으로 잘 렌더링되는지 확인.

## Phase 3: WebAssembly GNU Go 연동 (Web Worker + ccall API)

- **작업 내용:**
  1. `public/engine/gnugo.js` 와 `gnugo.wasm`을 사용해 `public/workers/gnugoWorker.js` 웹 워커 구축.
  2. 워커 내 통신 방식: GTP를 사용하지 않고, wasm-gnugo의 C API인 `ccall`을 직접 호출.
     - 메인 스레드 요청: `{ type: 'play', sgf: "(현재 게임의 SGF 문자열)" }`
     - 워커 내부 처리: `Module.ccall("play", "string", ["number", "string"], [0, sgf])`
     - 워커 응답 반환: 반환된 SGF 문자열을 파싱하여, 컴퓨터(GNU Go)가 방금 착수한 좌표만 뽑아내어 메인 스레드로 전송.
  3. 메인 스레드는 반환받은 좌표를 `tenuki.playAt(x, y)`에 입력하여 백돌 렌더링.
- **검증:** 유저 착수 후 SGF를 전달하면, 화면 프리징 없이 1~2초 내에 컴퓨터가 적절한 위치에 착수하는지 확인.

## Phase 4: 사활(Tsumego) 모드 구현

- **작업 내용:**
  1. `public/data/tsumego` 폴더에 샘플 SGF(바둑 기보 파일) 파일 3~5개 추가.
  2. **사활 페이지(`app/tsumego/page.tsx`)** 생성.
  3. SGF 파일을 파싱하여 특정 국면을 바둑판에 세팅.
  4. 유저가 돌을 두었을 때, 미리 정의된 정답 트리(Tree)와 비교하여 맞았는지 틀렸는지 판별.
  5. 정답 시 축하 모달 및 "다음 문제" 버튼 렌더링.
- **검증:** SGF 파일이 제대로 불려와서 사활 문제가 세팅되고, 정답/오답 처리가 정상적으로 작동하는지 확인.

## Phase 5: UI 폴리싱 및 효과음 적용

- **작업 내용:**
  1. 착수 시 '딱' 하는 바둑돌 소리 추가 (오디오 파일 활용).
  2. 승리, 패배, 사활 정답 시 `framer-motion`을 활용한 파티클 또는 팝업 애니메이션 추가.
  3. 모바일 기기(아이패드 등)에서 터치하기 편하도록 바둑판 마진 및 버튼 크기 미세 조정.
- **검증:** 모바일 뷰포트에서 레이아웃이 깨지지 않는지, 상호작용 피드백(소리/애니메이션)이 즉각적인지 테스트.
